Skill Publisher

MANDATORY skill for publishing ANY content to the knowledge portal — skills, walkthroughs, decisions, or templates. Use this skill whenever you have identified a reusable pattern worth preserving as a skill, or whenever a new skill/walkthrough has been added to the system by any means. Triggers on 'add this as a skill', 'save this as a skill', 'publish this skill', 'create a skill for this', 'add to the portal', 'new skill installed', 'create a walkthrough', 'add this to walkthroughs', 'publish this walkthrough'. This skill is also MANDATORY post-step: after ANY skill creation or installation, this workflow MUST run to completion — git commit + push + portal updates are not optional.

Use this skill to publish content to the GitHub Pages knowledge portal at gilesparnell/Claude. Covers skills, walkthroughs, and other portal content.

Portal URL: https://gilesparnell.github.io/Claude/ Repo: <claude-portal-repo> (local) / gilesparnell/Claude (remote)

Portal Structure

The portal has multiple content types, each with its own page and file location:

Content Type Portal Page Source Files How Cards Work
Skills docs/skills.html skills/<slug>/SKILL.md Auto-generated from frontmatter by GitHub Action
Walkthroughs docs/walkthroughs.html docs/diagrams/<slug>.html Manual — add card HTML + detail page
Templates docs/templates.html project-templates/ Manual

Shared design system: docs/portal.css — all pages import this. Never create custom CSS.

Part 1: Publishing a Skill

MANDATORY RULE

Every skill that is created, downloaded, installed, or modified MUST be:

  1. Committed to git — pushed to the gilesparnell/Claude repo
  2. Added to the knowledge portal — portal stub in <claude-portal-repo>/skills/<slug>/SKILL.md

This applies to ALL skill origins:

  • Skills you write from scratch
  • Skills installed from plugins or marketplaces
  • Skills downloaded or copied from external sources
  • Skills generated by any AI agent (Claude, Codex, etc.)
  • Skills modified or updated (commit the update + update portal stub version)

Two Formats — Always Create Both

1. Runtime version — ~/.claude/skills/<slug>/SKILL.md

What Claude reads when the skill is invoked. Full markdown content.

---
name: <slug>
description: Use this skill when... Triggers on '<phrase 1>', '<phrase 2>'.
---

# Skill Title

[Full instructions, principles, checklists, examples, gotchas]

2. Portal stub — <claude-portal-repo>/skills/<slug>/SKILL.md

What the GitHub Pages portal renders as a card. Frontmatter-only.

---
title: Display Title
scope: global | project
category: code | workflow | frontend | saas | tooling | engagement
icon: [emoji HTML entity]
description: One sentence. What it does and when to use it.
triggers:
  - phrase that invokes this
  - another trigger phrase
checks-label: Principles | Steps | Checks | Rules
checks:
  - Key point 1
  - Key point 2
  - Key point 3
  - Key point 4
version: 1.0
---

Scope Decision

Scope Location When
Global ~/.claude/skills/<slug>/ Useful on any project
Repo-specific <project>/.claude/skills/<slug>/ References a specific tech stack, client, or workflow

The portal stub always goes in <claude-portal-repo>/skills/<slug>/ regardless of scope.

Skill Step-by-Step

  1. Draft the runtime SKILL.md — write it as instructions to yourself: principles, checklist, examples, gotchas.
  2. Choose a slug — lowercase, hyphenated, descriptive. payment-webhooks not payments.
  3. Write the portal stub — 4 checks maximum. Pick the 4 most important points.
  4. Place the runtime file in the correct scope location (global or repo-specific).
  5. Place the portal stub in <claude-portal-repo>/skills/<slug>/SKILL.md.
  6. Commit and push (see "Git Push Protocol" below).
  7. Verify the skill appears in Claude Code's skill list (check system reminders).

Available CSS classes for content sections:

Class Use For
section-label Section heading (uppercase, small)
adr-grid + adr-card List of principle/rule cards with icon + title + desc
adr-num Left column of adr-card (emoji or number)
adr-body > adr-title + adr-desc Right column content
Inline <svg> Diagrams — use portal.css colour variables

Colour variables for SVG diagrams:

Variable Use For
--blue / --blue-bg / --blue-text / --blue-border Informational, neutral
--teal / --teal-bg / --teal-text / --teal-border Positive, terminal, code
--purple / --purple-bg / --purple-text / --purple-border Claude, AI, planning
--amber / --amber-bg / --amber-text / --amber-border Warning, result
--coral / --coral-bg / --coral-text / --coral-border Error, problem
--green / --green-bg / --green-text / --green-border Success, tmux

Dark mode: portal.css handles dark mode automatically via @media(prefers-color-scheme:dark). Use CSS variables, not hardcoded colours, in inline styles. For SVG diagrams, hardcoded fills are OK (they're self-contained).

Step 4: Commit and push

See "Git Push Protocol" below.

Git Push Protocol

This is the same for all portal content — skills, walkthroughs, decisions, templates.

cd <claude-portal-repo>
git pull --rebase origin main          # ALWAYS pull first
git add [files]                        # Stage specific files
git commit -m "feat: add <slug> [skill|walkthrough|decision]"
git push origin main

If the push is rejected (remote has newer commits):

git stash
git pull --rebase origin main
git stash pop
git push origin main

Do NOT force push. If conflicts arise, resolve them manually.

Completion Checklists

Skill Checklist

  • Runtime SKILL.md exists in ~/.claude/skills/<slug>/ (or repo-scoped equivalent)
  • Portal stub exists in <claude-portal-repo>/skills/<slug>/SKILL.md
  • Portal stub has correct frontmatter (title, scope, category, icon, description, triggers, checks, version)
  • Changes committed to gilesparnell/Claude repo
  • Changes pushed to origin/main
  • Skill appears in Claude Code's skill list (verify by checking system reminders)

Walkthrough Checklist

  • Detail page exists at <claude-portal-repo>/docs/diagrams/<slug>.html
  • Detail page uses portal.css (no custom CSS), has correct header/nav/footer
  • Card added to docs/walkthroughs.html BEFORE the empty state placeholder
  • Card has correct data-category, thumbnail, title, description, tags, link
  • Hero stats count updated in docs/walkthroughs.html
  • Changes committed to gilesparnell/Claude repo
  • Changes pushed to origin/main

If any of these are incomplete, the task is not finished. Do not move on to other work.

Gotchas

  • The description field in skill runtime frontmatter is what Claude Code uses for trigger matching — make it specific and include example trigger phrases in quotes
  • Don't make skills too broad — coding is useless; nextjs-api-routes is useful
  • If a skill is repo-specific, do NOT add a runtime version to ~/.claude/skills/ — it'll trigger incorrectly on other projects
  • Portal stubs with more than 4 checks look cluttered — curate ruthlessly
  • Keep the portal stub and runtime slug identical so they're easy to cross-reference
  • Walkthrough cards MUST go before the <!-- EMPTY STATE --> comment in walkthroughs.html
  • Walkthrough thumbnails should be visual (mini-diagrams), not plain text
  • Always use ../portal.css (relative path) in walkthrough detail pages
  • SVG diagrams can use hardcoded fills but inline styles should use CSS variables where possible
