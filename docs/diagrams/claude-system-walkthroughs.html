01

I add a new skill — how does it reach the knowledge portal?

A skill exists in two copies: the runtime one Claude actually reads (~/.claude/skills/) and the portal one in the public repo. The skill-publisher skill is the mandatory bridge between them — it fires after any skill is created or installed, whatever the origin. Once you push, everything downstream is automatic.

  1. Skill lands on your machine — written from scratch, installed from a plugin/marketplace, or generated by an agent. Runtime home: ~/.claude/skills/<slug>/SKILL.md.
  2. skill-publisher runs (Claude does this — it's a mandatory post-step). It creates the portal copy in the portal repo clone at skills/<slug>/SKILL.md with portal frontmatter plus the full body.
  3. Preview locally: npm test + npm run build in the clone.
  4. Commit + push to gilesparnell/Claude. A pre-commit scan blocks pricing / client / ADR terms — if it fires, stop and reassess, never bypass.
  5. Hands off from here: the build-portal.yml Action regenerates the index and the skill's detail page and auto-commits; GitHub Pages redeploys. Your card appears at gilesparnell.github.io/Claude/skills.html in ~a minute.

02

I want to redesign the knowledge portal's look & feel

Decided 23 Jul 2026: the Knowledge portal lives in gilesparnell/Claude only — never add Claude/ pages to the hub repo. The design contract (docs/DESIGN-HANDOVER.md) applies: Claude Design owns look-and-feel, Claude Code owns content. The trap to brief around: most portal pages are generated by the Action on every push — restyling the generated HTML gets overwritten within one push. The design must land in the stylesheet and the generator's template.

  1. Brief Claude Design with the repo (gilesparnell/Claude) and the three design surfaces: docs/portal.css (shared system, all pages import it), the hand-authored shells (docs/skills.html, walkthroughs.html, templates.html), and the HTML template inside scripts/build-docs-index.js. Say explicitly: never style docs/skills/* or docs/templates/* directly.
  2. Claude Design exports those files to an export folder (same pattern as today's site redesign).
  3. Claude Code pulls the export in, updates tests that assert the old markup, then runs npm test + npm run build — this regenerates every detail page under the new template, locally, before anything ships.
  4. Commit + push. The Action re-runs the same build in CI, auto-commits, Pages deploys.
  5. Verify live — curl the portal pages and eyeball skills, walkthroughs, templates and one detail page of each type.

03

I lose this laptop — rebuilding my Claude setup from GitHub

This is the strongest of the three stories, and since 23 Jul 2026 it is one command. A private ops repo is a live mirror of ~/.claude — global CLAUDE.md, identity files, skills, hooks, session-hooks, memory, settings — auto-synced at session end (verified byte-identical with your machine). Its scripts/recover.sh chains the whole restore; preview it any time with --dry-run.

  1. New machine: install Claude Code and the gh CLI, then gh auth login to the gilesparnell account.
  2. Run the one command — clone the private ops repo and run its scripts/recover.sh (the exact command is the first thing in that repo's SETUP.md). It runs install.sh (restores ~/.claude — rules, identity, 30+ skills, hooks, session-hooks, memory, settings), clones britfix from its own repo, and clones the working repos (hub site, knowledge portal). Covered by a 9-check test suite (scripts/recover.test.sh).
  3. Manual re-auth (OAuth can't be scripted — the script prints this checklist): Telegram plugin, Compound Engineering plugin, MCP servers (uvx workspace-mcp --single-user for Google Workspace, Supabase on first use).
  4. Smoke test: open a Claude session in a project and confirm it greets you with your rules (timezone, TDD, routing) and remembers project context.
