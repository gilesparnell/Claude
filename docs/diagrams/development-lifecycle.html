Walkthrough — Workflow
Development Lifecycle
Idea to Verified Feature
The complete skill pipeline: how every feature moves from brainstorm through TDD to browser-verified, shipped code. Enforced by hooks, not honour system.
The Pipeline
Enforcement Layer
The UserPromptSubmit hook in ~/.claude/settings.json injects mandatory instructions on every prompt. CLAUDE.md is advisory (can be forgotten). Hooks are deterministic (always fire). The hook hardcodes tdd-first and verification-before-completion as always-active when code is being produced.
1
Ideate
You have a vague idea, problem, or opportunity. Explore before committing.
Skills activated
/ce-brainstorm /last30days
/ce-brainstorm — collaborative dialogue to explore requirements, approaches, trade-offs, and scope before writing any code.
/last30days — (optional) research what the community is doing across Reddit, X, HN, YouTube. Grounds the brainstorm in real-world patterns.
Output
A requirements document or clear feature description, right-sized for the feature.
2
Plan
Requirements are clear. Break work into structured, ordered units with acceptance criteria.
Skills activated
/ce-plan
Transforms requirements into ordered implementation units with dependencies mapped, acceptance criteria per unit, and TDD flags on code units.
Output
A structured plan document (docs/plans/YYYY-MM-DD-NNN-*.md) with numbered units ready for execution.
3
Build (TDD)
Plan exists. Time to write code. The enforcement hook ensures TDD cannot be skipped.
Hook enforces: tdd-first ACTIVATED
Every prompt that produces code triggers mandatory TDD activation. Tests must be written and confirmed failing BEFORE implementation code exists.
Skills activated
/tdd-first (MANDATORY) /ce-work /security-review
Three test layers (all required for features)
Unit tests — every function with logic (3+ tests each: happy, sad, edge)
Integration tests — every boundary crossing (DB, API, services)
System/E2E tests — every user-facing feature (page renders, workflows complete)
4
Verify (E2E)
Code is built, TDD tests pass. Now verify the running application with a real browser + backend checks.
Hook enforces: verification-before-completion ACTIVATED
No completion claims without fresh verification evidence. "Should work" is not evidence. Run the command, read the output, then claim the result.
Skills activated
/e2e-feature-test /verification-before-completion (MANDATORY)
Test matrix categories
Happy path · Input validation · Edge cases · Error handling · Backend state · UI feedback · Permissions · Idempotency · Responsive
5
Ship
All tests green, E2E verified. Commit, push, create PR.
Actions
git commit → git push → gh pr create

Enforcement Architecture
Advisory (CLAUDE.md) vs Deterministic (Hooks)
CLAUDE.md — can be forgotten, per-conversation, honour system, context-dependent.
Hooks — always fire, survive restarts, enforced gates, unconditional.

The UserPromptSubmit hook in ~/.claude/settings.json runs ~/.claude/hooks/skill-forced-eval-hook.sh on every prompt. It injects mandatory instructions requiring skill evaluation before any work begins. This is what prevents TDD and verification from being forgotten.
