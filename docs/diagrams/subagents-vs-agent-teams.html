Walkthrough — Claude
Sub-agents &
Agent Teams
Two ways to put more than one Claude on a problem. They look similar and solve different problems. Here is how each one is wired, when to reach for it, and how to invoke it.
Where they sit in the stack

Agents don't work alone. Each layer feeds the one above it — persistent context at the bottom, event automation at the top. Sub-agents and Agent Teams sit in the middle: the part you actively orchestrate.

The core difference — hub-and-spoke vs. mesh

The single most useful mental model: a sub-agent setup is hub-and-spoke — one lead, workers that never talk to each other. An Agent Team is a mesh — peers that coordinate and message directly.

When to reach for Sub-agents
🔎
Parallel research
"Find every place X is used" across many directories — fan out, merge the summaries.
🧹
Protect the main context
A noisy 15-grep investigation runs in a worker and returns one clean paragraph.
🔍
Specialised one-shot pass
A security or performance review on a diff, then control comes straight back to you.
💰
Cheap volume
Run workers on a smaller model with scoped tools for repetitive, independent lookups.
When to reach for Agent Teams
⚖
Competing hypotheses
Several agents debug the same flaky test from different angles and compare notes.
👁
Multi-angle review
Security, performance, and conventions reviewers cross-check and reconcile findings.
🤝
Collaborative builds
Each teammate owns a slice of a feature and coordinates on the shared interface.
🛡
Red-team / blue-team
One proposes, another tries to break it; they iterate to a stronger answer.

How to invoke — Sub-agents

Define a custom worker as a markdown file in ~/.claude/agents/ (user-wide) or .claude/agents/ (project). The frontmatter scopes it; the body is its system prompt.

# 1 · Define a worker — ~/.claude/agents/cv-researcher.md
---
name: cv-researcher
description: Researches a role + company for interview prep.
            Use proactively when prepping for a role.
tools: Read, Grep, WebSearch, WebFetch
model: sonnet
---
You are an interview-prep research specialist. Given a role,
gather the company, the JD requirements, and likely questions.

# 2 · Invoke it
> Use the cv-researcher subagent to dig into this role.  # explicit
> Prep me for this job posting: ...            # auto-delegated
> Explore: where is auth handled?                # built-in, no file
How to invoke — Agent Teams

First enable the experimental flag in settings, then ask the lead to form a team and describe the angles — the teammates self-organise.

# 1 · Enable once — ~/.claude/settings.json
{ "env": { "CLAUDE_CODE_EXPERIMENTAL_AGENT_TEAMS": "1" } }

# 2 · Ask the lead to form a team
> Create a team to review this PR from three angles —
  security, performance, and conventions. Have them
  challenge each other and reconcile before reporting back.

> Put a team on this flaky test: each takes a different
  root-cause theory, then compare and converge.
⚠
Agent Teams is experimental
Its exact surface (dashboard, command names) still shifts across releases. The reliable invocation today is the flag plus a natural-language request to the lead — treat any specific CLI command for it as version-dependent.
They compose

These aren't either/or. A realistic flow stacks them: a hook fires on a new PR → the lead loads a review skill → it spins up an Agent Team for multi-angle review → each teammate dispatches its own sub-agents to fan out across the codebase → workers call MCP tools to pull GitHub and Supabase state → findings flow back up the mesh and the lead posts one reconciled summary.

Hook→ Skill→ Agent Team→ Sub-agents ×N→ MCP→ reconciled answer
