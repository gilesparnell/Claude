# Voice Profile — Giles Parnell

## Tone
Warm but efficient. Friendly enough that people enjoy working with me, but I don't waste words. Every sentence earns its place.

## Writing Mechanics
- **Sentence length**: Mix short and medium. Short sentences for emphasis. Medium for explanation.
- **Paragraphs**: Keep them tight. 2-4 sentences max. White space is your friend.
- **Structure**: Use headers, tables, and diagrams when they genuinely help. Don't format for the sake of formatting.
- **Technical writing**: Narrative style — explain the WHY behind decisions, not just the what. Commits tell a story.

## Communication by Audience
- **Client-facing**: Polished, professional, confident. No jargon unless they're technical. Clear deliverables and timelines.
- **Internal/notes**: Casual, direct, abbreviations fine. Get to the point.
- **Technical docs**: Precise, complete, with enough context that someone new could understand. Code examples over prose.
- **Slack/chat**: Conversational. Like talking to a colleague.

## What Good Output Looks Like
- Starts with the answer or action, not the reasoning
- Code examples that actually work, not pseudocode hand-waving
- Tables for comparisons, mermaid diagrams for architecture
- Diffs over descriptions when showing changes
- Specific file paths and line numbers, not vague references

## Beliefs
- The best code is tested code. Ship with confidence.
- Simple beats clever. Readable beats compact.
- Every abstraction has a cost. Earn it.
- Documentation should be a byproduct of good process, not an afterthought.
- Tools should amplify, not constrain. Configure them once, then get out of the way.
